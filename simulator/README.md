# E-Commerce Simulator

Generates realistic order and payment data for the dbt Snowflake training project.

## Overview

The simulator models an e-commerce order lifecycle as a probabilistic state machine:

```
placed → confirmed → shipped → delivered
                  ↘ cancelled  ↘ returned
```

It runs in three modes and writes to either DuckDB (local/default) or Postgres (production).

## Quick Start

### 1. Bootstrap — load reference data from CSV files

```bash
python -m simulator --bootstrap
```

Reads `samples/customers.csv`, `samples/addresses.csv`, and `samples/products.csv`
(configured via `bootstrap.csv_dir` in `config.yaml`) and creates the full schema.

### 2. Historical — generate compressed order history

```bash
python -m simulator --historical 12
```

Generates 12 months of order history (volume set by `simulation.num_orders` in `config.yaml`).
Run `--bootstrap` first.

### 3. Stream — emit live orders at real-time pace

```bash
# Run indefinitely (Ctrl-C to stop)
python -m simulator --stream

# Run for a bounded window (useful for testing)
python -m simulator --stream --duration 300
```

Loops continuously, inserting one order per `tick_interval` seconds. Injects new customers
at a random rate drawn from `Uniform(0, new_customer_rate_max)` per order (see ADR-016).
`--duration SECONDS` stops the loop cleanly after that many wall-clock seconds.

## Database

By default the simulator uses **DuckDB** — no server required, data stored in `simulator.duckdb`.

To switch to Postgres, update `config.yaml`:

```yaml
database:
  type: postgres
  # dsn is read from DATABASE_URL env var
```

Then set `DATABASE_URL=postgresql://user:pass@host:5432/dbname`.

## Configuration

All tunable parameters are in `simulator/config.yaml`:

| Key | Default | Description |
|---|---|---|
| `database.type` | `duckdb` | `duckdb` or `postgres` |
| `database.path` | `simulator.duckdb` | DuckDB file path |
| `bootstrap.csv_dir` | `samples` | Directory containing seed CSVs |
| `simulation.num_orders` | `50000` | Orders generated per `--historical` run |
| `simulation.new_customer_rate_max` | `0.10` | Upper bound for new-customer injection rate |
| `simulation.max_retries` | `3` | Max payment retries before terminal failure |
| `simulation.tick_interval` | `1` | Seconds between orders in stream mode |

## Temporal Consistency

The simulator enforces two temporal invariants in historical data:

1. **Orders never predate their customer.** `--historical` uses
   `max(window_start, customer.created_at)` as the earliest possible order date
   for each customer. Customers created after `end_date` are excluded entirely.

2. **Lifecycle events are monotonically ordered from order_date.** Each state
   transition advances a logical clock by a realistic delay:
   placed→confirmed (0–30 min), confirmed→shipped (1 hr–3 days),
   shipped→delivered (2–7 days), delivered→returned (1–30 days).
   `orders.updated_at` reflects the timestamp of the final event.

Orders placed near the end of a historical window will have in-flight lifecycle
events (e.g., shipped but not yet delivered) — this is intentional and the right
data to test dbt incremental models against.

## Bootstrap Data

Seed CSVs are generated by `samples/simulator_base_data.ipynb`. To regenerate:

```bash
# Run from project root
jupyter nbconvert --to notebook --execute --output simulator_base_data.ipynb samples/simulator_base_data.ipynb
```

Current volumes (tunable in the notebook):

| File | Rows |
|---|---|
| `addresses.csv` | 150,000 |
| `customers.csv` | ~255,000 |
| `products.csv` | 25,000 |

## Testing

```bash
pytest simulator/tests/
```

Tests run against real in-memory DuckDB — no mocks, no server required.
